# 架构决策记录（ADR）

格式：每条一节，包含 `上下文 / 选项 / 决策 / 后果`。新决策追加在文件末尾，不修改历史决策。

## ADR-0001 后端 Web 框架选型

- **日期**：2026-06-18
- **上下文**：DeepMoat 后端从 FastAPI 迁移到 Rust，需要选 Web 框架
- **选项**：
  - axum（tokio 官方，tower 生态，类型友好）
  - actix-web（性能强，actor 模型）
  - poem（API-first，对 OpenAPI 友好）
- **决策**：**axum**
- **后果**：
  - + 与 tokio / tower 生态一致，中间件复用
  - + 编译期路由类型检查
  - − 中文资料偏少，但英文社区活跃

## ADR-0002 数据库

- **日期**：2026-06-18
- **上下文**：旧版同时存在 `stock.db`(SQLite) 与生产 PG；目标统一
- **决策**：**PostgreSQL**（不回退 SQLite）
- **后果**：
  - + 支持窗口函数 / JSONB / 物化视图，护城河计算更顺手
  - + sqlx 编译期校验需要在线 DB 或离线 schema 缓存
  - − 本地开发需要 docker compose 起 PG

## ADR-0003 前端方向

- **日期**：2026-06-18
- **上下文**：用户明确"前端方向不改"
- **决策**：**保留 Vue 3 + Vite + UnoCSS**，仅做 UI 美化（UI 库升级、design tokens、深色模式、ECharts）
- **后果**：
  - + 不重写业务逻辑，工作量可控
  - + 现有路由 / API 调用层零改动
  - − 美化空间受现有组件结构限制

## ADR-0004 数值计算库

- **日期**：2026-06-18
- **上下文**：旧版重度用 pandas
- **选项**：polars / ndarray + 手写
- **决策**：**polars**
- **后果**：
  - + 表达式 API 与 pandas 接近，迁移直观
  - + 性能显著优于 pandas
  - − 二进制体积偏大；护城河引擎里有些复杂逻辑可能仍需 itertools 风格手写

## ADR-0005 项目战略转向（P1-P9 全栈重构 → P0-P5 渐进升级）

- **日期**：2026-06-18
- **上下文**：与用户深入对话后，确认本项目的真实定位是
  - 个人长期投资工具（用户单人用，不对外管钱）
  - 用户目标：公司内部转岗后端 / Web，Rust 是学习投资
  - 时间投入：工作日 1-2h + 周末 12h，约 13h/周
  - 用户的痛点不是"Python 性能不够"，而是 ① 投资体系不成熟 ② skills 散乱 ③ 移动端无法访问 ④ AI 能力没用够 ⑤ UI 乱
- **选项**：
  - A. 原计划 P1-P9 全栈 Rust 重构（工程价值低，学习投入大，DeepMoat 主线被阻塞）
  - B. 完全不做 Rust（转岗故事弱）
  - C. **渐进升级 + Rust 切片** — Python 主体增量改进 + 把"护城河引擎"独立成 Rust 微服务作为转岗故事
- **决策**：**C**
- **后果**：
  - + DeepMoat 主线 6 个月内可达到生产可用 + 移动可访问 + 自动化运行的状态
  - + Rust 切片是一个完整、独立、可演示的工程，适合面试讲故事（架构决策 / 工程实现 / 性能数据）
  - + Python 后端持续可用，不阻塞用户日常投资决策
  - − 用户需要接受"DeepMoat 不会成为 100% Rust 项目"
  - − 跨语言集成有维护成本，需要清晰的 HTTP API 边界

## ADR-0007 Rust 切片选定护城河引擎

- **日期**：2026-06-18
- **上下文**：Rust 切片需要一个"清晰边界 + CPU bound + 业务自洽"的模块作为转岗故事载体
- **候选**：护城河引擎 / Tushare 抓取 / 财务指标计算 / AI 估值适配器
- **决策**：**护城河引擎**（`app/service/moat_engine.py`）
- **理由**：
  - 输入清晰（财务三表数据）
  - 输出清晰（评分对象）
  - 无副作用（纯计算，不写 DB）
  - 是 DeepMoat 的核心差异化（品牌名都叫 DeepMoat）
  - 算法复杂度足够展示工程能力（5 维评分 + ROIC/WACC + 持续性 + 量化）
  - 性能对比有戏（批量算 5000 只票时，Rust 优势可量化）
- **不选其他的原因**：
  - Tushare 抓取：IO bound，Rust 优势小
  - 财务指标：依赖现成 DataFrame 库，polars 在 Python 里也能用
  - AI 估值适配器：主要是网络请求 + JSON，Rust 优势小
- **后果**：
  - + 故事完整：架构决策（为什么切这块）+ 工程实现（完整后端栈）+ 性能数据（实测）
  - + 切片大小可控：核心算法不超过 1000 行 Rust 代码
  - − 需要保证 Rust 实现与 Python 实现行为一致（通过 golden_data 对照）
