# 架构决策记录（ADR）

格式：每条一节，包含 `上下文 / 选项 / 决策 / 后果`。新决策追加在文件末尾，不修改历史决策。

---

## ADR-0001 后端 Web 框架选型

- **日期**：2026-06-18
- **上下文**：DeepMoat 后端从 FastAPI 迁移到 Rust，需要选 Web 框架
- **选项**：
  - axum（tokio 官方，tower 生态，类型友好）
  - actix-web（性能强，actor 模型）
  - poem（API-first，对 OpenAPI 友好）
- **决策**：**axum**
- **后果**：
  - + 与 tokio / tower 生态一致，中间件复用
  - + 编译期路由类型检查
  - − 中文资料偏少，但英文社区活跃

## ADR-0002 数据库

- **日期**：2026-06-18
- **上下文**：旧版同时存在 `stock.db`(SQLite) 与生产 PG；目标统一
- **决策**：**PostgreSQL**（不回退 SQLite）
- **后果**：
  - + 支持窗口函数 / JSONB / 物化视图，护城河计算更顺手
  - + sqlx 编译期校验需要在线 DB 或离线 schema 缓存
  - − 本地开发需要 docker compose 起 PG

## ADR-0003 前端方向

- **日期**：2026-06-18
- **上下文**：用户明确"前端方向不改"
- **决策**：**保留 Vue 3 + Vite + UnoCSS**，仅做 UI 美化（UI 库升级、design tokens、深色模式、ECharts）
- **后果**：
  - + 不重写业务逻辑，工作量可控
  - + 现有路由 / API 调用层零改动
  - − 美化空间受现有组件结构限制

## ADR-0004 数值计算库

- **日期**：2026-06-18
- **上下文**：旧版重度用 pandas
- **选项**：polars / ndarray + 手写
- **决策**：**polars**
- **后果**：
  - + 表达式 API 与 pandas 接近，迁移直观
  - + 性能显著优于 pandas
  - − 二进制体积偏大；护城河引擎里有些复杂逻辑可能仍需 itertools 风格手写
